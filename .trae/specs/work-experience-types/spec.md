# 工作经历类型功能 - 产品需求文档

## 概述
- **Summary**: 为工作经历添加三种可选类型（学习、实习、工作），在管理后台支持选择类型，并在前台展示时根据类型显示不同的图标和样式
- **Purpose**: 让工作经历的展示更加丰富和有层次，用户可以区分不同阶段的经历
- **Target Users**: 网站管理员和访客

## 目标
1. 在数据库模型中添加经历类型字段
2. 在管理后台表单中添加类型选择器
3. 在前台时间线中根据类型显示不同的图标和样式
4. 更新前后端API以支持类型字段

## 非目标（范围外）
- 不添加更多的经历类型（如成就等）
- 不修改现有的数据库迁移脚本
- 不添加经历类型的自定义功能

## 背景与上下文
当前系统中工作经历只有一种类型，前台组件 `WorkExperienceTimeline.tsx` 已经预留了类型配置（work、education、achievement），但后端模型和管理后台还没有支持。用户现在要求添加三种类型：学习、实习、工作。

## 功能需求
- **FR-1**: 数据库模型支持经历类型字段
- **FR-2**: 管理后台表单支持选择经历类型
- **FR-3**: 前台时间线根据类型显示不同的图标和样式
- **FR-4**: API接口支持传递和返回类型字段

## 非功能需求
- **NFR-1**: 默认值为"工作"类型，保持向后兼容
- **NFR-2**: 类型选择为下拉单选，用户体验友好
- **NFR-3**: 前台显示时，不同类型有明显的视觉区分

## 约束
- **技术约束**: 使用现有的技术栈（Go/Gin, React/TypeScript, PostgreSQL）
- **业务约束**: 仅支持三种固定类型：学习、实习、工作
- **依赖约束**: 需要数据库迁移来添加新字段

## 假设
- 现有数据的类型字段默认为"工作"
- 前台展示的图标和颜色可以基于现有配置调整
- 类型字段为可选字段，向后兼容

## 验收标准

### AC-1: 数据库模型支持类型字段
- **Given**: 数据库已运行
- **When**: 应用启动并执行自动迁移
- **Then**: work_experiences表应包含type字段，默认值为'work'
- **Verification**: programmatic
- **Notes**: 使用GORM的AutoMigrate功能

### AC-2: 管理后台支持类型选择
- **Given**: 管理员已登录并进入工作经历管理页面
- **When**: 添加或编辑工作经历时
- **Then**: 表单中应显示类型下拉选择器，包含"学习"、"实习"、"工作"三个选项
- **Verification**: human-judgment

### AC-3: 前台展示不同类型的经历
- **Given**: 存在不同类型的工作经历数据
- **When**: 访问首页查看工作经历时间线
- **Then**: 不同类型的经历应显示对应的图标和样式
- **Verification**: human-judgment

### AC-4: API支持类型字段
- **Given**: API服务正常运行
- **When**: 创建、更新或查询工作经历
- **Then**: 请求和响应中应包含type字段
- **Verification**: programmatic

### AC-5: 向后兼容
- **Given**: 存在没有类型字段的旧数据
- **When**: 查询和展示旧数据
- **Then**: 旧数据应默认按"工作"类型处理
- **Verification**: programmatic
